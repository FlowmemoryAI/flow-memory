# Repository Agent Instructions

## Diagrams and architecture output

- For architecture, infrastructure, agent-network, protocol, API, wallet, payment, settlement, credential, or data-flow work, include Mermaid diagrams in planning notes, docs, and final reports when useful.
- Prefer at least one `flowchart` and one `sequenceDiagram` for new subsystems.
- Keep Mermaid node IDs simple ASCII identifiers.
- Never include secrets, API keys, private keys, seed phrases, personal data, raw private memory, or credential material in diagrams.
- Final implementation reports should include: architecture diagram, request or interaction sequence when applicable, changed files, validation results, and remaining blockers.
- When using OMP, enable `tools > render mermaid` if available so Mermaid diagrams render in turn outputs.
