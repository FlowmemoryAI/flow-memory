# Storage

## Purpose

Storage makes agent state, audit trails, manifests, memory records, and economy transitions durable without requiring Redis, Qdrant, Neo4j, or a live chain. SQLite is the default local persistence baseline; optional external stores remain adapter seams.

## Implemented behavior

Status: functional local prototype.

- `src/flow_memory/storage/sqlite_store.py` owns schema bootstrap and the schema version table.
- `src/flow_memory/storage/agent_store.py` persists agent profiles and state snapshots.
- `src/flow_memory/storage/event_store.py` persists runtime events.
- `src/flow_memory/storage/audit_store.py` persists audit events and now supports hash-chained audit appends through `append_chained()`.
- `src/flow_memory/storage/marketplace_store.py` persists marketplace tasks, bids, escrows, settlements, disputes, and slashing events.
- `src/flow_memory/storage/reputation_store.py` persists DID-bound reputation updates.
- `src/flow_memory/storage/memory_store.py` persists memory records.
- `src/flow_memory/storage/skill_store.py` persists skill manifests.
- `src/flow_memory/storage/export.py` exports table contents to JSONL.
- `src/flow_memory/storage/replay.py` builds deterministic replay chains and verifies stored audit hash chains.

## Audit replay and tamper evidence

Hash-chained audit records store:

- `chain_index`
- `previous_hash`
- `payload_hash`
- `chain_hash`

`AuditStore.verify_chained()` replays the chain in order and fails closed on duplicate IDs, missing chain fields, broken indexes, broken previous hashes, payload tampering, or chain-hash mismatch. This is local tamper evidence, not a substitute for external notarization or write-once storage.

## API snapshot artifacts

`docs/API_SNAPSHOT.json` is generated by:

```bash
python scripts/export_api_snapshot.py --write docs/API_SNAPSHOT.json
```

The snapshot records endpoint count, paths, operations, manifest hash, and OpenAPI hash for CI/release drift checks.

## Signed local checkpoints

`src/flow_memory/storage/checkpoints.py` can sign a replay result summary with the local development signing seam. The checkpoint covers the latest audit hash and event count. This is useful for local release checks, but production use still requires real key custody and external checkpoint publication.

Replay and checkpoint a SQLite audit database with:

```bash
python scripts/replay_audit_log.py --db flow-memory.sqlite3 --checkpoint --require-events
```


## Limitations

- SQLite is local persistence, not a distributed or high-availability database.
- Hash chaining detects tampering after the fact; it does not prevent a compromised host from deleting or rewriting the whole database.
- External Redis/Qdrant/Neo4j adapters remain optional seams.
- Backup, encryption-at-rest, retention, and multi-process operational policy still need production hardening.
